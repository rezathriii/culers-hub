# Culers Hub

A self-hosted Telegram bot that tracks Barça news from multiple RSS feeds and posts updates to a channel — with clean summaries generated by a local LLM, smart deduplication, and zero third-party API costs.

Built to keep [@alldaysbarca](https://t.me/alldaysbarca) updated without lifting a finger.

---

## What it does

Every 30 minutes the bot checks five Barça news sources. New articles are cleaned, summarised by a locally-running Qwen2.5 model, and posted to your Telegram channel. Each post includes the source name, publication date, a 2–3 sentence summary, and a photo when the feed provides one.

A few things it handles quietly in the background:

- **No duplicates.** Articles are stored in a local SQLite database after being sent. Same story from two different sources? The second one gets dropped.
- **No old news.** Only articles published in the last hour pass through. The first run won't flood your channel with weeks of backlog.
- **Resilient summaries.** If the LLM is unavailable, posting continues normally using the original RSS excerpt.

## Requirements

- Docker + Docker Compose
- A Telegram bot token (free, from [@BotFather](https://t.me/BotFather))
- A Telegram channel where the bot is admin
- _(Optional)_ A [llama.cpp](https://github.com/ggerganov/llama.cpp) server running locally for LLM summaries

## Setup

### 1. Create a Telegram bot

1. Open [@BotFather](https://t.me/BotFather) → `/newbot` → follow the prompts → copy the token.
2. Add the bot as an **admin** of your channel with permission to post messages.
3. Find your channel's chat ID:
   - Public channel: `@your_channel_username` works directly.
   - Private channel: forward any message from the channel to [@userinfobot](https://t.me/userinfobot) — it'll return the numeric ID (e.g. `-1001234567890`).

### 2. Configure

```bash
cp .env.example .env
```

Open `.env` and fill in at minimum `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID`. Everything else has sensible defaults.

### 3. Run

```bash
docker compose up -d --build
```

```bash
docker compose logs -f   # watch live output
docker compose down      # stop
```

## Configuration

All settings live in `.env`. The only required ones are the two Telegram credentials.

| Variable                 | Default                 | Description                                     |
| ------------------------ | ----------------------- | ----------------------------------------------- |
| `TELEGRAM_BOT_TOKEN`     | —                       | **Required.** Token from @BotFather             |
| `TELEGRAM_CHAT_ID`       | —                       | **Required.** `@channel` or numeric ID          |
| `CHECK_INTERVAL_MINUTES` | `30`                    | How often to poll all feeds                     |
| `MAX_ARTICLE_AGE_HOURS`  | `1`                     | Ignore articles older than this. `0` = no limit |
| `SEND_INTERVAL_SECONDS`  | `60`                    | Pause between consecutive Telegram posts        |
| `LLM_ENABLED`            | `true`                  | Set to `false` to skip LLM and use raw RSS text |
| `LLM_BASE_URL`           | `http://localhost:9000` | Base URL of your llama.cpp server               |
| `LLM_TIMEOUT`            | `30`                    | Seconds to wait for LLM before falling back     |

## LLM summaries (optional)

The bot uses a locally-running [llama.cpp](https://github.com/ggerganov/llama.cpp) server to rewrite each article excerpt into a clean 2–3 sentence summary. No API key, no cost, no data leaving your server.

Tested with **Qwen2.5 3B Q4_K_M** on a Hetzner CX43 (8 vCPU / 16 GB RAM, no GPU). Each summary takes ~5 seconds on CPU.

To run the server as a persistent systemd service:

```bash
# Create /etc/systemd/system/llama-server.service pointing to your binary + model
systemctl enable --now llama-server
```

The Docker container reaches the host server via `host.docker.internal:9000` (configured automatically in `docker-compose.yml`).

If you don't want LLM summaries, set `LLM_ENABLED=false` in `.env` — the bot falls back to the RSS excerpt and nothing else changes.

## How deduplication works

Two independent layers prevent the same story from being posted twice:

1. **ID-based** — every sent article's unique feed ID is stored in SQLite (persisted on a Docker volume). Survives restarts.
2. **Title similarity** — within each cycle, article titles are compared using Jaccard token similarity. If two titles from different sources share ≥50% of their meaningful words, the second one is skipped. This also catches stories already sent in previous cycles.

The cycle log shows what happened:

```
Cycle complete – sent: 4  skipped (seen): 22  skipped (duplicate title): 2  total: 28
```
